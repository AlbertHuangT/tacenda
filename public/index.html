RSA Encrypted Chat

end-to-end encrypted  ·  zero server-side state  ·  no accounts

How it works

The server is a router, not a vault. It holds an in-memory table mapping public keys to active WebSocket connections. When you send a message, the server forwards an encrypted blob to the recipient's socket — and nothing else. It cannot read the content. The decryption key never leaves your device.

Each page load generates a fresh RSA-2048 key pair via the browser's Web Crypto API. Your public key becomes your address on the network. When you close the tab, the private key is destroyed. The server keeps no logs, no message history, no persistent identifiers of any kind.

Routing is address-based: you paste a contact's public key, and messages are delivered to it. The first message includes your own public key, so the recipient can reply without a second round of copy-paste.

Encryption

RSA-OAEP alone cannot encrypt arbitrary-length data — the ceiling for a 2048-bit key is around 190 bytes. The system uses hybrid encryption:

  1. A random AES-GCM-256 key and 12-byte IV are generated per message.
  2. The plaintext is encrypted with AES-GCM, producing an authenticated ciphertext.
  3. The AES key is wrapped with the recipient's RSA-2048-OAEP public key.
  4. Three base64 fields — k, iv, msg — are transmitted.

The server forwards these three fields verbatim without inspecting them. To the server, every message is indistinguishable from noise.

Long-term identity  — optional

Session keys are ephemeral by design. For recurring contacts, you can generate a persistent identity key pair and exchange your public key with a friend once — offline, through any channel you trust.

To start a session, broadcast a handshake: your current session public key, encrypted with your contact's long-term public key. Every connected client receives the broadcast and silently attempts decryption. Only the intended recipient succeeds. The server never learns who is signaling whom.

Identity key operations are CLI-only. Web clients execute JavaScript delivered by the server at runtime — a compromised server could inject code that exfiltrates a loaded private key. A compiled CLI binary is auditable, static, and immune to that class of attack.

Limitations

The server sees routing metadata: which public key is sending to which public key, and when. It cannot read message content, but it can observe traffic patterns — who talks to whom and how often.

There is no sender authentication in the ephemeral web chat. Anyone who obtains your contact's public key can send messages that appear to come from a new conversation partner. The CLI's handshake protocol mitigates this for identity-key users.

CLI

# build from source
cd cli && go build -o rsa-chat-cli .

# generate a long-term identity key pair (run once, keep the file safe)
./rsa-chat-cli --keygen --out ~/.rsa-chat/identity.pem

# connect to the server (replace URL with your own if self-hosting)
./rsa-chat-cli --key ~/.rsa-chat/identity.pem \
               --server wss://ppb1s0n.us.kg/ws

# find a contact — paste their long-term public key
> /find MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8A...

# accept an incoming handshake and start chatting
> /accept 1
> hello

No account registration. No server-side key storage. Disconnect and every session key is gone. The identity key file never leaves your machine.
